介绍

在 Web 上，我们有一个非常有效的应用程序协议，它为世界各地的计算机进行有意义的交换做好准备：超文本传输协议，或 HTTP。HTTP 是 TCP/IP 之上的应用程序级协议，是一种通信协议。在教授网页设计和开发时，HTTP似乎经常被遗忘，这是一种耻辱：理解它将帮助您定义更好的用户交互，实现更好的网站性能，并创建有效的工具来管理Web上的信息。

WHAT

通信协议

在了解具体细节之前，让我们考虑一个基本的通信场景。为了能够进行通信，两方（无论是软件、设备、人员等）需要：

  • 语法（数据格式和编码）
  • 语义（控制信息和错误处理）
  • 定时（速度匹配和排序）

当两个人见面时，他们会使用通信协议进行交流：例如，在日本，一个人会用身体做出特定的手势。其中一种手势是鞠躬，这是用于交互的语法。在日本习俗中，鞠躬的手势（除其他外）与问候某人的语义有关。最后，当一个人向另一个人鞠躬时，在特定时间内在两人之间建立了一系列事件

通信协议

在线通信协议包含相同的元素。

语法将是字符序列，例如我们用于编写协议的关键字。

语义是与这些关键字中的每一个相关联的含义，

最后，时间是两个或多个实体交换这些关键字的顺序。

HTTP在机器中的什么位置？

HTTP本身运行在其他协议之上。当连接到网站时，例如在 ，用户代理正在使用 TCP/IP 协议套件。TCP/IP 模型设计于 1970 年，具有 4 个不同的层：www.example.org

  1. 链接描述了对物理介质的访问（例如使用网卡）
  2. Internet 描述了数据的信封和路由 — 数据的打包方式 （IP）
  3. 传输描述了数据从起点到最终目的地（TCP、UDP）的传递方式
  4. 应用程序描述传输消息 （HTTP） 的含义或格式

HTTP 是位于通信协议之上的应用程序协议。记住这一点很重要。将模型划分为独立的层有助于发展平台的各个部分，而无需重写所有内容。例如，TCP（一种传输协议）可以在不修改 HTTP（应用程序协议）的情况下发展。在实践中，当我们努力实现高性能通信时，细节会变得有点丑陋。在前几篇 HTTP 文章中，我们将重点介绍 TCP/IP 模型中定义的层分离。HTTP被定义为通过HTTP消息在两个软件之间传递信息。我们塑造和设计这些消息的方式对客户端（例如浏览器（网站）及其中介（代理）有影响。

让我们到达服务器

端口 80 是用于连接到 Web 服务器的默认端口。我们可以自己使用 shell 来尝试。打开终端/命令行，然后尝试使用以下命令打开与端口 80 的连接：www.opera.com

​ telnet www.opera.com 80

让我们来谈谈 HTTP

让我们再次尝试与服务器通信。在终端/命令行中输入以下消息：

​ telnet www.opera.com 80

完成此操作并建立通信后，快速键入以下 HTTP 消息（在连接自动关闭之前），然后按两次 Enter/Return：

​ GET / HTTP/1.1

​ Host: www.opera.com

此消息指定：

  • GET：我们希望信息的表示。GET
  • /：我们想要获取的信息存储在网站的根目录下。
  • HTTP/1.1：我们说的是 1.1 版。HTTP
  • Host:：我正在尝试访问特定站点。
  • www.opera.com：站点的名称为 www.opera.com。

HOW

在 Web 上获取内容

请记住，HTTP 客户端（浏览器）始终向 HTTP 服务器发起请求。HTTP 协议为客户端提供了一些工具来向服务器表达其意图：URI、HTTP 方法和 HTTP 标头。

给事物起个名字

URI 是 Web 的基石之一。他们解决了整个互联网上存在的一个重要问题：如何唯一地识别网络上的某些东西。

现在，如果你让别人拿起一个物体并把它带到你面前，没有太多的解决方案。您使用的单词将开始识别您需要的对象。

  • 你可以问朋友：“把书给我”
  • 朋友可能会回答：“好的，但是哪一个呢？
  • 你：“另一个房间里的书。
  • 朋友（去另一个房间）：“嗯？哪一个？
  • 你（有点恼火）：“绿色的！
  • 朋友：“嘿……冷静。这里有两本绿皮书！

你站起来，终于自己拿到了书。但所有这些都可以简单得多。我们如何唯一地识别我们以后可能想要访问的这些东西？记忆是一种可能性。我们如何向某人发出指示以获得正确的对象？让我们为它创建一个系统：

  1. 剪纸或使用黄色便签。
  2. 在你周围的房间或你的桌子上，把空的笔记放在你以后需要访问的物品（例如书籍）附近或上面。
  3. 在每个笔记上，输入一个唯一的 ID。
  4. 在另一个空间或另一张桌子上重复该过程。

​ 请记住，它不仅供您自己使用，而且可以进行交互（并且显然可以避 免您的挫败感）。我只是为我做了。我在纸上写了以下标识符，每个 对象都附有以下标识符。

​ myRoom.org/table/book/001

​ myRoom.org/shelf/book/002

​ otherRoom.org/cup/001

​ otherRoom.org/flower/001

​ otherRoom.org/book/001

我想现在你已经明白了。我们现在有一个标签系统，用于精确识别空间中的事物。在 Web 上，我们正在处理一个信息空间，标识此信息的标签是 URI。

在 Web 上，客户端会向服务器发送更多信息（HTTP 标头），以帮助协商 HTTP 请求。然后，服务器将能够根据这些标头调整其答案。Opera Dragonfly 中有一个非常实用的工具，用于创建自定义 HTTP 请求并检查服务器 HTTP 响应：您可以在“发出请求”选项卡的“网络”部分下找到它。

![screenshot20231101 (1)](screenshot20231101 (1).png)

在“提出请求”选项卡中，有三个区域：

  • 网址：资源标识符（或 Web 地址）
  • 请求正文：客户端将发送到服务器的内容。（“发送请求”按钮实际上会通过网络将请求发送到服务器。)
  • 响应：发出请求后服务器的响应。

让我们自定义 HTTP 请求

  1. 在 URL 部分中复制https://www.opera.com/
  2. 将以下 HTTP 请求复制并粘贴到网络选项卡的“请求正文”部分。我们将标头添加到前面的最小 HTTP 请求中。Accept-Language
  3. 然后点击“发送请求”按钮。

​ GET / HTTP/1.1

​ Host: www.opera.com

​ Accept-Language: en

如果我们请求一种服务器上不存在的语言，会发生什么情况？试试吧：例如中文：zh

​ GET / HTTP/1.1

             Host: www.opera.com

​ Accept-Language: zh

你拿到了英文版的网站——不解吗？这是网站设计选择的一部分。服务器HTTP响应可以以不同的方式设计：它可以简单地回答“不，我们没有这个地址的中文版本”或“这个地址没有中文版本，这里有我们可用的语言”（带有指向不同语言的特定版本的链接列表）。但Opera UX部门决定，当请求语言未知时，服务器将默认发送英文版本。这确实是一个选择问题：这里没有正式的标准。

这就是为什么我经常向UX人员和前端开发人员传授一些HTTP知识。HTTP 是一种用于管理客户端和服务器之间交互的应用程序协议。因此，了解它的工作原理将有助于在为人类和机器（机器人、API 客户端等）创建网站时设计有意义的交互。
